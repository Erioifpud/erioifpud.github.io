对Vue事件机制错误理解的纠正

发表于 2019-04-01 | 更新于 2019-06-18 | 评论数：
本文字数： 2.5k | 阅读时长 ≈ 2 分钟

如果你觉得\$emit发出事件能在父组件中使用\$on捕获，那我觉得你应该看下去。

先说结论，\$emit只会在当前实例触发事件，\$on只会捕获当前实例的事件。

on和emit的运行机制

先看一段源代码：

这是$emit函数的实现，首先是通过事件名 event 在_events属性中找到事件的回调函数，如果回调函数存在，那么就执行回调函数。

在这个过程中 vm 为 this，也就是说在组件中使用this.$emit()的时候，vm 是指向组件本身的。

那么_events属性是从哪来的呢？接着看一段$on的源代码：

关键在 else 块中的第一行，当调用$on函数时，会将事件名 event 作为一个键，放入_events对象中，值为回调函数组成的数组（由于可以多次使用$on注册事件回调，所以回调函数都会被存入数组中，按注册顺序执行）。

概括

$on注册的回调函数会被放入当前实例 vm的_events对象中，$emit会在当前实例vm的_events对象中找事件的回调函数去执行。

所以$emit和$on这两个函数涉及的对象都是vm，和父组件毫无关系。

关于v-on

首先，v-on指令是放在元素或组件中的，在指令的定义函数中的el是属于当前组件或元素的，这没毛病。

v-on:click="handleClick"这种写法会将handleClick函数作为binding.value传给指令定义函数，这也没毛病。

父级模板里的所有内容都是在父级作用域中编译的；子模板里的所有内容都是在子作用域中编译的。

这是官方文档里的话，那么现在组件 parent 的模版是这样的：

这是组件 parent 的模版，那很明显这里的handleClick函数是来自于组件parent的（使用 parent 的作用域），组件 child 上的v-on:click会监听child的事件（因为 el 来自于 v-on 挂载的组件），所以v-on:click="handleClick"这个语句的意思实际上是：

监听child组件上的click事件，触发事件后调用parent提供的handleClick函数（listener是在组件child上的）

而不应该被理解成组件 parent 去监听 child 的 click 事件，满足条件就触发 handleClick。

再进一步

刚刚对于 v-on 的结论还是有些不准确，结合$emit的行为来看，v-on 实际上是将父组件提供的handle函数放入当前组件vm的_events中，于是vm.$emit()，会找到父组件传入的handle函数并执行。
